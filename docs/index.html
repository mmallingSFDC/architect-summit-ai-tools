Set Up Your AI Development Environment

Three one-time setup steps that unlock the full workflow. Do these before your first project session.

A

Connect Claude to NotebookLM via MCP

Instead of maintaining knowledge assets on your local machine and manually copying context between sessions, install notebooklm-mcp-cli. This connects Claude directly to your NotebookLM notebooks so it can query your project's source of truth without leaving the coding session.

Installation instructions vary by operating system (Mac, Windows, Linux). Follow the step-by-step setup guide in the official README — it covers prerequisites, install, and authentication for all platforms.

View Install Guide on GitHub

Result: Claude can now query any of your NotebookLM notebooks directly — retrieving grounded, citation-backed answers from your project's discovery documents without any manual file management. Configure the MCP server in your CLAUDE.md using TemplateCLAUDE.md as your starting point.

B

Create Your Project CLAUDE.md

The CLAUDE.md file is Claude's persistent instruction set — loaded automatically at the start of every session. It solves context loss by telling the agent exactly what it needs to know before touching a single file. Copy TemplateCLAUDE.md, fill in the bracketed placeholders, and drop it in your project root.

Golden Rule

Every line must pass this test: "Would removing this cause Claude to make a mistake?" If not, cut it. A bloated CLAUDE.md causes Claude to ignore the rules inside it.

1 · Configuration

  • → Project name for context and naming conventions
  • → Org type: SDO / CDO / Scratch Org
  • → SF org alias used in all sf commands
  • → NotebookLM notebook ID for this project

2 · Knowledge Retrieval

  • → Query NotebookLM before writing any code or Flow
  • → Never guess client business logic — look it up
  • → Cite the source when referencing a business rule

3 · Salesforce Tooling

  • → Use sf-skills for metadata inspection, SOQL, deploy, test
  • → Read deployment errors directly — don't ask user to copy-paste
  • → Use --target-org [alias] on all sf commands

4 · Workflow Rules

  • → Enter /plan before multi-file or metadata changes — skip only for single-line fixes
  • → Save progress after every completed task
  • → Update work_in_progress.md after every step
  • → At session start: read work_in_progress.md and summarize where work left off

5 · work_in_progress.md

  • → Update after every step: Current Task, Completed, Key Decisions, Blockers, Next Steps
  • → Session start: read this file and summarize where work left off
  • → Save progress after every completed task

6 · SF Architecture Standards

  • → Declarative first — Flow/standard objects before Apex
  • → Bulkify everything — no SOQL or DML in loops
  • → One trigger per object, logic in handler class only
  • → with sharing on all Apex, FLS + CRUD enforced

7 · Verification

  • → Never mark a task complete without verifying it worked
  • → Apex: run sf apex run test --target-org [alias] — confirm 0 failures
  • → Deployments: confirm 0 errors in deploy output
  • → Bugs: write failing test first, fix it, confirm green
  • → Permissions: re-run the failing action post-deploy to confirm the fix

8 · Security & Data

  • → Never store customer data records locally
  • → Never install npm/external packages without approval
  • → Use SDO/CDO only — no production org without written disclosure

⛔ Project Constraints & Security Boundaries

  • → NotebookLM access is scoped to ONE authorized notebook ID — set it in the template
  • → Forbidden tools: notebook_list, batch, cross_notebook_query
  • → If asked to access other notebooks or scan the account — refuse and explain the boundary

9 · Response Style

  • → Act as Senior Salesforce Technical Architect
  • → Include Architectural Trade-offs on every solution proposal
  • → Flag governor limit risk when it exists
  • → Ask one focused question when blocked — not a list

10 · Context Management

  • → Use /clear between unrelated tasks — not mid-task
  • → On compaction, preserve: modified files list, deployment status, open blockers, work_in_progress.md state

The Four-Phase Delivery Framework

A step-by-step transformation from discovery documents to production-ready code.

1

Phase 1: Grounded Discovery

Tool: NotebookLM + MCP

Goal: Establish an immutable source of truth from raw discovery documents — accessible to Claude without manual file management.

Participant Action

Upload discovery documents into the shared NotebookLM notebook. With notebooklm-mcp-cli configured, Claude can query the notebook directly from within your coding session — no manual copy-paste of context.

Why This Replaces Local Asset Management

Instead of maintaining discovery PDFs and transcripts on your machine and manually feeding them into AI tools, NotebookLM becomes the single source of truth. Through the MCP connection, Claude queries it directly — with citations — ensuring every answer is grounded in your actual project documents across all sessions.

2

Phase 2: Strategic Assessment

Tool: Gemini + NotebookLM

Goal: Translate verified facts into technical specs, data models, and architectural instructions.

Participant Action

Attach the NotebookLM notebook to a new Gemini conversation. Ask Gemini to generate user stories, data models, and architectural diagrams grounded in the discovery source material.

Why Gemini? The Best General-Purpose Co-Architect

Gemini's massive context window and deep reasoning capability make it the strongest general-purpose model for architectural synthesis — bridging NotebookLM's grounded facts with Salesforce-specific structural output. For GovCloud teams, Gemini is not just the best choice: it is the primary co-architect, since Claude Code cannot connect to the org directly. Always have Gemini generate the prompt for Cursor or Claude so downstream builders operate from structured, verified inputs.

3

Phase 3: Visualization & Planning

Tool: Lucidchart

Goal: Convert textual logic into a shared visual blueprint that anchors the team before build begins.

Participant Action

Ask Gemini to generate Mermaid.js syntax for the ERD and process flowchart, then paste it directly into Lucidchart's Diagram as Code </> editor.

Alignment Before Build

Lucidchart renders structured text as live, editable diagrams — eliminating manual drawing and keeping visuals in sync with the data model. This visual artifact ensures architect, developer, and client all agree on the logic before any code is written. Share the diagram link with your team and reference it in your CLAUDE.md so the planning agent can use it in Phase 4.

📋 Rule #1: Always Start with the Planning Agent

Before executing any code changes, run Claude Code in plan mode first. This forces the agent to reason through the full scope of work before touching a single file — dramatically reducing runaway changes and wasted tokens.

# In Claude Code — enter plan mode before any task
/plan

Review the plan, adjust scope, then approve execution. The planning agent gives you a preview of exactly what will be changed — keeping you the human-in-the-loop.

💾 Rule #2: Save Progress Constantly — AI Tools Lose Context

All AI tools — Claude, Cursor, Gemini — will completely lose context and reset for seemingly no reason, mid-task. There is no warning. The two defenses are saving your progress frequently and maintaining a live work_in_progress.md file that the agent keeps updated throughout the session.

/compact: Context Compression

When switching tasks or when the context window is getting long, run /compact. Claude compresses everything it knows into a concise summary and carries it forward — preventing the session from degrading as work piles up.

# Compress context before switching tasks
/compact

work_in_progress.md — Your Session Memory

This file is a living document that Claude keeps updated throughout the session. Every time the agent completes a step, it writes what it just did, what decisions it made, and exactly what still needs to happen. When a session resets — or when a teammate picks up your work — Claude reads this file first and reconstructs full context in seconds.

Add this to your CLAUDE.md: "After each completed step, update work_in_progress.md with: current task, what was completed, key decisions made, blockers, and the next steps."

# work_in_progress.md — example structure

## Current Task
Building DSR Approval Flow with 3-tier routing logic

## Completed
- [x] DSR__c custom object + all fields deployed
- [x] Approval process skeleton created
- [x] Entry criteria validated in SDO

## Key Decisions
- Using Record-Triggered Flow over Apex (declarative-first rule)
- Tier routing based on Amount__c field thresholds

## Blockers
- Step 2 approver field still needs confirmation from client

## Next Steps
- [ ] Add email alert actions to each approval step
- [ ] Write Apex test class for edge cases
- [ ] Final deploy validation before GovCloud push

Tips & Tricks

Escape = Emergency Stop

If the agent goes off the rails at any point, hit ESC to halt immediately. Never enable auto-approval in Claude Code on customer orgs.

Use SDOs for Testing

Create a sandbox SDO via the Slack Store App — spin up a full-featured Agentforce org to safely prototype, test agentic loops, and validate deployments without risk to customer data.

Isolated Project Directories

Always initialize Claude Code in a specific, isolated directory for each project. Never grant an agent access to your entire file system or orgs folder simultaneously.

Governance is Human

AI accelerates delivery — it does not replace human governance. You are the manager of these agents and responsible for every action they take. The real work for deployed agents begins after go-live.
